# forward-panel v1.3.8

修两个 v1.3.7 暴露的问题(都是 master 改动,agent 无需重编):

## 1. 节点「中转不可达·已舍弃」徽标误判 → 改用入口视角判健康

**问题**:v1.3.7 的 `relayHealthLoop` 由 **master 自己 tcping** 出口的中转端口判可达。但 master 与入口处在不同网络位置——aws 的中转端口从入口深圳拨得通(分段检测深圳→aws 隧道全绿),master 却可能拨不通(安全组只放行入口、或国内外路由不同),于是错标"不可达"。正如你指出的「有些 IP 国内外是两种状态」,**可达性是相对入口的,不是相对 master 的**。

**修复**:`probeRelays` 不再 master 自拨,改为**汇总各入口对每个出口中转地址的 tcping 历史**(与分段检测同一份、入口视角的数据)判健康:
- 有入口最近探到成功 → 健康(纳入轮询、无徽标)。
- 有入口在探它且最近全失败 → 不可达(移出轮询、标"已舍弃")。
- 没有任何入口探它(无判据)→ 保持乐观、不翻转。

→ aws 深圳→aws 隧道既然全绿,徽标会自动消失、aws 留在出口轮询。只有当入口真的探不通某出口时才标记,vantage 正确。

(`tcpReachable` 仍由边缘容灾 `edgeHealthLoop` 使用,不受影响。)

## 2. 新建规则的出口段"暂无数据" → 规则下发后刷新出口/入口 ProbeAdd

**问题**:`dispatchRuleApply`(新建/改规则)只对**入口**发 `rule.apply`,不给出口节点重发 `sync`。而出口探落地、入口探隧道的目标(ProbeAdd)只随 `sync` 下发。所以新建带出口的规则后,出口拿不到落地地址 → 出口→落地段一直"暂无数据";入口也拿不到隧道地址(出口群组规则)。只有 master 重启(全节点重注册)才偶然刷新。

**修复**:`dispatchRuleApply` 末尾,**规则带出口(exit_node/exit_group)时触发 `redispatchAll`** —— 给所有在线节点重发 sync(含最新 `nodeProbeExtra`),出口立刻拿到落地、入口拿到各出口隧道地址,分段检测即有数据。

## 关于 anytls 规则(hk2)"真实不通"——大概率不是协议 bug

结合现象推断:
- hk2 直连(入口深圳→落地)2ms 通 → 落地 `203.0.113.13:888` **从深圳可达**。
- hk2 过任意出口都不通 → 很可能落地 **从出口 aws/狗妈咪 不可达**(与 rule 11 落地不同,后者出口可达所以通)。
- 这正是「同一 IP 国内外两种状态」:落地对入口可达、对出口不可达。与 anytls 协议本身无关,是**落地↔出口的可达性**。

**本版修好出口探测后**,hk2 的出口→落地段会直接显示 aws/狗妈咪 能否到 `203.0.113.13:888`:
- 显示"连接失败" → 印证出口到不了落地,这就是 hk2 不通的原因(换个出口能到的落地,或让落地放行出口 IP)。
- 显示有延迟(绿) → 那才需要进一步查 anytls 隧道协议层(届时把现象告诉我再深挖)。
